ElAtaba (العتبة) Marketplace

System Architecture, DbContext Schema & API Endpoint Specifications

1. System Architecture Definitions

The ElAtaba backend is built using Clean Architecture principles in .NET 10. This ensures that the core domain logic is decoupled from frameworks, databases, and external presentation APIs.

[Elattaba.API] (Presentation / Entry Point)
       │
       ▼ (Depends on)
[Elattba.Application] (Use Cases, Validation & Logic)
       │
       ▼ (Depends on)
[Elattba.Core] (Domain Entities, Enums, and Interfaces)
       ▲
       │ (Implements Core Interfaces)
[Elattba.InfraStructure] (EF Core, Repositories, ONNX AI matcher)

1.1 Presentation Layer (Elattaba.API)

  • Role: Represents the HTTP REST API interfaces and acts as the project entry point.
  • Responsibilities:
    • Exposes controllers and handles URL routing.
    • Enforces security policies (Authorization headers, HttpOnly cookies, CORS policies).
    • Registers Swagger API definitions for document rendering.
    • Executes rate-limiting checks and global exception filters.

1.2 Application Layer (Elattba.Application)

  • Role: Dictates the functional use cases and workflows of the system.
  • Responsibilities:
    • Implements orchestrating services (e.g., OrderService, ProductService, CheckoutService).
    • Performs client parameter validation using FluentValidation validators.
    • Configures business mapping and DTO translation operations.

1.3 Core / Domain Layer (Elattba.Core)

  • Role: Represents the absolute center of the Clean Architecture model, holding no dependencies on other layers or database libraries.
  • Responsibilities:
    • Defines database entities (e.g., Product, Store, Order, Review).
    • Declares database abstraction contracts (e.g., IRepository<T>, IUnitOfWork).
    • Defines global enumerations (e.g., Role, OrderStatus) and data models.

1.4 Infrastructure Layer (Elattba.InfraStructure)

  • Role: Provides technology-specific implementations for interfaces declared in Core.
  • Responsibilities:
    • Implements Entity Framework Core repositories and manages context mappings.
    • Hosts the in-memory ONNX AI Image Embedding matching service using MobileNetV2.
    • Manages physical disk operations for files and product images.

2. Double DbContext Schema Details

To guarantee a clean separation between account security credentials and commercial business logic, the application utilizes a Double DbContext architecture mapping to separate databases.

  [AppIdentityDbContext]                         [El3atbaDbContext]
  (ASP.NET Core Identity DB)                 (Marketplace Commercial DB)
  ┌──────────────────────┐                    ┌────────────────────────┐
  │ - AspNetUsers        │                    │ - Users (Profiles)     │
  │ - AspNetRoles        │                    │ - Stores               │
  │ - AspNetUserRoles    │                    │ - Products             │
  │                      │                    │ - Orders               │
  └──────────┬───────────┘                    └───────────▲────────────┘
             │                                            │
             └───────────── Sync Key: DomainUserId ───────┘

2.1 Context Specifications

  • Identity Context (AppIdentityDbContext): Inherits from IdentityDbContext<AppUser, IdentityRole<int>, int>. It is responsible for handling login hashes, passwords, and security claims via standard ASP.NET Identity tables (e.g., AspNetUsers, AspNetRoles).
  • Commercial Context (El3atbaDbContext): Inherits from DbContext. It manages the core e-commerce engine, holding tables like Users, Stores, Products, Orders, OrderItems, and Reviews.

2.2 Decoupled Profiles & Synchronization Flow

The system decouples identity security credentials from market profile records. They are linked via a mapping key:

  • Sync Key (DomainUserId): The Users profile table in the commercial database has a column called DomainUserId. This maps directly to the primary key Id in the identity AspNetUsers database table.
  • Registration Flow:
    1. The registration endpoint receives credentials and user profile fields.
    2. It creates a security account in the Identity database using UserManager<AppUser>.
    3. Once successful, the generated AppUser.Id (integer) is captured.
    4. The system then inserts a record into the commercial database's Users table, mapping DomainUserId = AppUser.Id to connect both database entities.
  • Updates Flow: When modifying personal profile details (like First Name and Last Name), the backend updates both databases under an orchestrating routine, syncing changes to both DbContext contexts.

3. Swagger API Endpoint Specifications

Below are the RESTful API endpoints exposed by the backend services:

Controller HTTP Method Endpoint Route Authorization Level Description / Purpose
Account POST /api/Account/register Public Registers a new user profile with selected Buyer/Seller role.
Account POST /api/Account/login Public Authenticates credentials and sets HTTP-Only JWT tokens.
Account POST /api/Account/logout Public Revokes active session tokens and clears JWT cookies.
User GET /api/User Admin Only Retrieves all registered user profiles on the platform.
User GET /api/User/{id} Authorized Gets profile details of a specific user.
User PUT /api/User/{id} Authorized Updates profile data (name, phone, governorate, address).
Store GET /api/Store Public Lists all registered active vendor stores.
Store GET /api/Store/{id} Public Gets details and metadata for a specific storefront.
Store POST /api/Store Seller Only Creates and registers a new store.
Store PUT /api/Store/{id} Seller Only Updates storefront metadata and configurations.
Store DELETE /api/Store/{id} Admin Only Permanently deletes a store registration.
Product GET /api/Product Public Lists catalog products (supports paging, search, and category).
Product GET /api/Product/{id} Public Gets detail records for a product (includes volume discount tiers).
Product POST /api/Product Seller Only Lists a new wholesale product listing under a store.
Product PUT /api/Product/{id} Seller Only Updates specifications, details, or stock levels.
Product DELETE /api/Product/{id} Seller/Admin Removes a product listing from the platform.
AI Search POST /api/ProductImage/upload Seller Only Uploads product photos and triggers AI index generation.
AI Search POST /api/ProductImage/search-by-image Public Runs cosine vector matching to find products via uploaded query image.
AI Search POST /api/ProductImage/rebuild-embeddings Admin Only Manually rebuilds the database vector embeddings index.
Checkout POST /api/Checkout/submit Buyer Only Processes cart orders, splits orders by store, and deducts inventory.
Order GET /api/Order Authorized Lists orders (gated by caller role: Admin/Buyer/Seller).
Order GET /api/Order/{id} Authorized Gets invoice items and delivery details for an order.
Order PUT /api/Order/{id}/status Seller/Admin Updates fulfillment state (e.g., Confirmed, Shipped, Delivered).
Review POST /api/Review Buyer Only Submits a purchase review (gated to completed, delivered orders).
Governorate GET /api/Governorate Public Retrieves shipping locations in Egypt.
Governorate POST /api/Governorate Admin Only Registers a new governorate shipping location.
Shipping Rate GET /api/ShippingRate Public Retrieves shipping cost breakpoints.
Shipping Rate POST /api/ShippingRate Admin Only Configures carrier rates for specific governorates.
Message GET /api/Message/thread/{otherUserId} Authorized Retrieves chat message history.
Message POST /api/Message/send Authorized Sends a message to another user thread.

4. Git Flow & Branching Strategy Summary

  • 🔴 main: Stable production-ready branch. Direct pushes are disabled.
  • 🔵 development: Team integration branch. Used as integration and staging branch.
  • 🟢 feat/*: Feature-specific workspaces (e.g., feat/multi-store-checkout, feat/ai-image-search). Run dotnet test (C#) and npm run build (React) locally before merging.
