# 🚀 ElAtaba — Frontend Implementation Scope

منصة **ElAtaba** هي Multi-Vendor Marketplace متكاملة، بتدعم 4 أنواع مستخدمين: **Buyer, Seller, StoreManager, Admin**. الوثيقة دي بتغطي كل حاجة محتاجها فريق الـ Frontend لبناء الواجهة، بما فيها آخر تحديث خاص بـ **Multi-Store Checkout**.

## 1. التأسيس الهيكلي (App Shell & Core)

بناء هيكل التطبيق بحيث يدعم التوجيه الذكي وحالات التحميل والأخطاء:

- **Layouts:** تصميم Navbar و Sidebar يتغيران ديناميكياً حسب الـ `Role` والـ `StoreId`.
- **State Management:** إدارة حالات الـ Loading, Error, Auth State, Toast Notifications.
- **Route Guards (حماية المسارات):**
  | نوع المسار | الوصول |
  |---|---|
  | Public Routes | تصفح المنتجات، البحث |
  | Buyer Routes | سلة المشتريات، الطلبات السابقة |
  | Seller Routes | إدارة المتجر والمنتجات |
  | Admin Routes | لوحة تحكم الإدارة |
- **API Client (Axios/Fetch):**
  - ضبط الـ Base URL مركزياً.
  - ⚠️ **مهم جداً:** تفعيل `withCredentials: true` لضمان إرسال الـ HttpOnly Cookies في حالة الويب.
  - قراءة الـ Token المحفوظ وإرساله في `Authorization: Bearer` كخطوة احتياطية (Hybrid Auth).

## 2. إدارة الهوية (Authentication & Authorization)

النظام يعتمد على **Hybrid JWT** (يعمل عبر الـ Cookies للويب والـ Headers للموبايل).

### Register
`POST /api/Account/register`
- **الحقول:** `Email`, `Password`, `FirstName`, `LastName`, `Phone`, `Role`, `GovernorateId`, `City`, `ShippingAddress`
- لازم تستدعي `GET /api/Governorate` الأول عشان تملأ قائمة المحافظات.
- خيار "Admin" **لا يظهر** في واجهة التسجيل العادية.

### Login
`POST /api/Account/login`
- **الحقول:** `Email`, `Password`
- **الرد:** `Token`, `UserId`, `Email`, `Role`, `StoreId`
- **آلية الحفظ:** الباك إند بيحقن الـ Token في HttpOnly Cookie تلقائياً. الفرونت يحفظ بس `UserId`, `Role`, `StoreId` في الـ LocalStorage أو الـ State للتحكم في الواجهة.

### Logout
`POST /api/Account/logout`
- استدعاء الـ Endpoint (بيمسح الكوكي من المتصفح)، بعدها مسح الـ State المحلية والتوجيه لصفحة الدخول.

## 3. مصفوفة الصلاحيات (UI Permissions Matrix)

| الدور | الصلاحيات |
|---|---|
| **Public Visitor** | تصفح المنتجات، العروض، البحث بالصور، والمتاجر — بدون أي لوحة تحكم |
| **Buyer (0)** | عرض المنتجات، إضافة للسلة، إتمام الطلب، متابعة الطلبات، إضافة تقييمات (Reviews) |
| **Seller (1) / StoreManager (3)** | لوحة تحكم البائع: إضافة/تعديل المتجر، المنتجات، العروض، أسعار الجملة، تحديث حالة الطلبات الخاصة بمتجره فقط |
| **Admin (2)** | لوحة تحكم الإدارة: المستخدمين، التصنيفات، المحافظات، شركات الشحن، تحديث الذكاء الاصطناعي |

## 4. تجربة التسوق والكتالوج (Public Catalog)

### الرئيسية
`GET /api/Product`
- لازم يتم إبراز: صورة المنتج، **السعر الحالي** (`CurrentPrice`)، **السعر القديم** (`OldPrice`)، ونسبة الخصم.

### تفاصيل المنتج
`GET /api/Product/{id}`
- عرض الصور المتعددة، تفاصيل المتجر، وجدول أسعار الكميات (Pricing Tiers).

### البحث الذكي بالصور
`POST /api/Product/search-by-image`
- **UI Feature:** شاشة مخصصة لرفع صورة (`multipart/form-data`) وعرض المنتجات المشابهة (Matched Products) مع نسبة التطابق (Score).

## 5. سلة المشتريات والدفع (Cart & Multi-Store Checkout) 🛒

> تم تحديث هذه الآلية لتدعم الشراء من متاجر متعددة في عملية دفع واحدة.

### السلة (Cart UI)
- يُسمح للمستخدم بإضافة منتجات من متاجر مختلفة.
- **التجميع البصري (Visual Grouping):** المنتجات في السلة تظهر مقسمة تحت اسم كل متجر (مثال: قسم "متجر الأمل" وتحته منتجاته، ثم قسم "متجر النور" وتحته منتجاته).
- **التسعير الديناميكي:** تحديث السعر الفردي للمنتج فوراً في السلة لو زادت الكمية ودخلت في شريحة خصم جديدة (Pricing Tiers).

### الدفع (Checkout UI)
إجمالي الفاتورة لازم يوضح:
```
إجمالي المنتجات + إجمالي الشحن (مجموع شحن كل المتاجر) − الخصومات = المبلغ النهائي
```

### إنشاء الطلب (Bulk Checkout)
`POST /api/Order`
- بيتم إرسال **طلب واحد** يحتوي على: عنوان الشحن، طريقة الدفع (`Cash=0`, `Online=1`)، ومصفوفة بكل المنتجات `OrderItems`.
- الباك إند يقسّمها تلقائياً لطلبات منفصلة (Order لكل متجر) ويرجّعها للفرونت.

## 6. لوحة تحكم البائع (Seller Dashboard)

### إدارة المتجر
لو `StoreId = null` بعد تسجيل الدخول → اعرض زرار واضح **"Create Your Store"** (`POST /api/Store`).

### إدارة المنتجات (`multipart/form-data`)
- دعم رفع الصور أثناء إنشاء المنتج: `POST /api/Product`
- أو إنشاء منتج مع عرض حصري مباشرة: `POST /api/Product/create-with-offer`

### العروض (Offers)
واجهة تدعم نوعين:
- عرض يشمل المتجر بالكامل: `appliesToAllProducts: true`
- عرض على منتجات معينة: `productIds: [1, 2]`

### طلبات المتجر (Seller Orders)
- عرض الطلبات الخاصة بمتجره فقط.
- تحديث حالة الطلب: `PUT /api/Order/{id}/status` (راجع [قيم الـ Enums](#12-ملحق-قيم-الـ-enums))

## 7. لوحة تحكم الإدارة (Admin Dashboard)

شاشات إدارة بسيطة (CRUD) للموارد الأساسية:

| المورد | العمليات |
|---|---|
| التصنيفات (Categories) | `GET`, `POST`, `PUT`, `DELETE` |
| المحافظات وشركات الشحن (Governorates & Carriers) | إدارة بياناتهم + التسعير الخاص بكل محافظة (Shipping Rates) |
| البحث بالذكاء الاصطناعي | زرار مخصص للأدمن بس: `POST /api/ProductImage/rebuild-embeddings` لإعادة بناء فهارس البحث بالصور |

## 8. هيكل البيانات وإدارة الأخطاء (Data Shape & Error Handling)

### شكل الاستجابة (API Wrapper)
أغلب الـ Responses بتيجي بالهيكل ده:
```json
{
  "statusCode": 200,
  "message": "Success",
  "data": { }
}
```
⚠️ الفرونت لازم يقرأ البيانات من `response.data.data`.

### اعتراض الأخطاء (Interceptors)

| Status Code | السلوك المطلوب |
|---|---|
| `400 Bad Request` | عرض رسالة الخطأ القادمة من الباك إند (Validation/Business Error) |
| `401 Unauthorized` | مسح الـ State والتوجيه لصفحة الـ Login |
| `403 Forbidden` | عرض شاشة "لا تملك صلاحية لهذه الصفحة" أو إخفاء العنصر |
| `500 Server Error` | رسالة خطأ عامة |

## 9. التحقق من صحة البيانات (Client-Side Validations)

لازم تتفعّل الـ Validation قبل إرسال أي Request:

| الحقل | القاعدة |
|---|---|
| السعر (`price`) | `> 0` |
| المخزون (`stock`) | `>= 0` |
| الكمية (`quantity`) | `> 0` |
| الخصم (`discount`) | `> 0` و `<= 100` |
| تواريخ العروض | `startDate < endDate` |
| تقييم المنتج (`rating`) | من `1` إلى `5` |
| السلة | مينفعش Checkout لو الـ Items فاضية |

## 10. سيناريو العرض التوضيحي (Demo Flow) 🎬

فريق الفرونت لازم يجهز السيناريو ده للتجربة:

1. "زائر" يتصفح الموقع ويشوف العروض والمنتجات.
2. يستخدم ميزة "البحث بالصورة" ويشوف تطابق المنتجات.
3. "بائع" يسجل دخول، يضيف متجر جديد، يرفع منتج مع صوره.
4. البائع يضيف "عرض خصم" على المنتج.
5. "مشتري" يسجل دخول، ويلاحظ تغيّر سعر المنتج وظهور السعر المشطوب ونسبة الخصم.
6. المشتري يضيف منتجات من متجرين مختلفين في السلة ويدفع مرة واحدة.
7. الطلبات تظهر مقسمة حسب المتجر في لوحة تحكم المشتري.
8. المشتري يستلم الطلب ويضيف تقييم (Review).

## 11. ملاحظات للواجهة الأمامية (Backend TODOs & Context)

- الـ **Pagination** و الـ **Advanced Filtering** للـ `GetAll` قيد التطوير حالياً في الباك إند.
- الـ **Refresh Tokens** هتتضاف لاحقاً لضمان بقاء جلسة المستخدم.
- ⚠️ رفع الملفات `multipart/form-data`: المتصفح بيحط هيدر الـ Boundary تلقائياً — **متضبطش الـ Content-Type يدوياً** في كود الـ Axios/Fetch عشان الملفات متتلفش.
